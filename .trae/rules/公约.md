# **《Android 商业级架构 AI 开发公约》v1.0**

> **适用范围**：本项目所有 AI 辅助编程会话（Trae CN / Qoder / Copilot 等）\
> **生效时间**：立即生效，不可协商\
> **更新方式**：需架构师本人确认后方可修改

## **第一部分：核心架构层级铁律**

### **1.1 模块分层定义**

text

```
┌─────────────────────────────────────────────────────────────┐
│  app (壳工程)                                              │
│  职责：Application 入口、Splash、Manifest 配置、混淆配置    │
│  依赖：所有 features、service、middleware                   │
├─────────────────────────────────────────────────────────────┤
│  features (业务功能层)                                     │
│  职责：具体业务功能（登录、首页、个人中心、支付...）        │
│  依赖：core、service、middleware                           │
│  约束：禁止 features 之间互相依赖                         │
├─────────────────────────────────────────────────────────────┤
│  middleware (中间件层)                                     │
│  职责：路由(router)、权限(permission)                      │
│  依赖：core                                                │
│  约束：禁止依赖 service 层                                 │
├─────────────────────────────────────────────────────────────┤
│  service (基础服务层)                                      │
│  职责：启动(startup)、崩溃(crash)、日志(logger)            │
│  依赖：core                                                │
├─────────────────────────────────────────────────────────────┤
│  core (核心基建层)                                         │
│  职责：纯基础能力，不包含任何业务逻辑                       │
│  子模块：                                                  │
│  ├── base      - BaseActivity/ViewModel、UiState          │
│  ├── network   - Retrofit、OkHttp、拦截器链               │
│  ├── storage   - Room、MMKV、KVStorage (唯一存储门面)      │
│  ├── cache     - LruCache、MemoryCache                     │
│  ├── common    - 纯 Kotlin 工具 (日期/字符串/加密/正则)    │
│  └── ui        - 通用 UI 组件 (LoadingMask/骨架屏/State)   │
└─────────────────────────────────────────────────────────────┘
```

### **1.2 依赖方向规则（绝对不可逆）**

kotlin

```
// ✅ 正确的依赖方向（由下至上）
features → middleware → service → core
features → core
features → service
features → middleware

// ❌ 绝对禁止
core → features           // 基建层绝不能依赖业务层
core → service            // 基建层绝不能依赖服务层
core:common → core:storage // 纯工具层绝不能依赖存储层
service → features         // 服务层绝不能依赖业务层
middleware → features      // 中间件层绝不能依赖业务层
```

### **1.3 `core:common` 禁飞区（最高优先级）**

**规则**：`core:common` 模块是项目的“数学层”，必须保持绝对纯净。

kotlin

```
// ❌ 严禁在 core:common 中导入以下任何类
import android.content.Context
import android.content.SharedPreferences
import com.tencent.mmkv.MMKV
import androidx.room.Room
import android.util.Log

// ✅ 允许在 core:common 中导入
import kotlinx.datetime.*
import java.security.MessageDigest
import java.util.regex.Pattern
import kotlin.text.Regex
```

**验证标准**：`core:common` 必须能在纯 JVM 环境下运行单元测试，不加载任何 Android 或 Native 库。

## **第二部分：模块职责细化规范**

### **2.1 存储模块 (`core:storage`) —— 唯一存储出口**

**核心原则**：全项目只有一个 KV 存储门面，禁止泄露第三方库细节。

kotlin

```
// ✅ 正确的对外接口
object KVStorage {
    fun putString(key: String, value: String, scope: Scope = Scope.USER)
    fun getString(key: String, defaultValue: String = "", scope: Scope = Scope.USER): String
    fun putInt(key: String, value: Int, scope: Scope = Scope.USER)
    fun getInt(key: String, defaultValue: Int = 0, scope: Scope = Scope.USER): Int
    fun remove(key: String, scope: Scope = Scope.USER)
    fun clear(scope: Scope = Scope.USER)
    enum class Scope { USER, CONFIG, CACHE }
}

// ❌ 严禁将 MMKV 对象暴露给上层
fun getUserKV(): MMKV  // 绝对禁止！
```

**执行规则**：

- 删除 `core:common` 中的 `PreferenceUtil`
- 删除 `core:storage` 中的 `MMKVHelper`
- 业务层统一使用 `KVStorage`，不直接接触 MMKV

### **2.2 网络模块 (`core:network`) —— 唯一 API 出口**

**核心原则**：所有网络接口定义在 `core:network` 中，业务模块直接使用。

kotlin

```
// ✅ 正确的数据模型定义位置
// core/network/src/main/java/.../model/UserResponse.kt
data class UserResponse(val id: String, val name: String, val avatar: String)

// ✅ 业务模块直接使用 Network 层的 DTO
// features/feature_login/.../LoginViewModel.kt
val result = apiService.login(phone, code)  // 返回 UserResponse

// ❌ 严禁在业务模块复制粘贴定义新的 UserInfo 类
data class UserInfo(val id: String, val name: String) // 绝对禁止！
```

**执行规则**：

- 所有 API 返回的 JSON 数据类（DTO）定义在 `:core:network/model`
- 业务模块**直接复用**这些 DTO，禁止创建同名类

### **2.3 基类 (`core:base`) —— 纯洁性公约**

**核心原则**：Base 类只处理通用框架能力，不包含任何业务逻辑。

kotlin

```
// ✅ BaseViewModel 只包含状态机、协程封装、通用异常处理
abstract class BaseViewModel<State, Event> : ViewModel() {
    protected abstract val _state: MutableStateFlow<State>
    val state = _state.asStateFlow()
    protected suspend fun execute(block: suspend () -> State) { ... }
    protected fun sendEvent(event: Event) { ... }
}

// ❌ 严禁在 Base 类中添加业务特定方法
class BaseActivity {
    fun showLoginDialog() { ... }     // 禁止！这是登录模块的职责
    fun getUserLocation() { ... }     // 禁止！这是定位模块的职责
    fun startPayment() { ... }        // 禁止！这是支付模块的职责
}
```

**执行规则**：

- 新增功能时，优先使用**组合（Composition）** 而非继承
- 如需共享功能，创建独立的工具类或委托类

## **第三部分：编码规范强制规则**

### **3.1 资源文件规范**

**规则**：所有 UI 文本必须定义在 `strings.xml` 中。

xml

```
<!-- ❌ 严禁在布局文件中硬编码 -->
<TextView android:text="登录" />

<!-- ✅ 必须引用 strings.xml -->
<TextView android:text="@string/login_title" />

<!-- ❌ 严禁在 Kotlin 代码中硬编码 -->
Toast.makeText(context, "登录成功", Toast.LENGTH_SHORT).show()

<!-- ✅ 必须引用 strings.xml -->
Toast.makeText(context, R.string.login_success, Toast.LENGTH_SHORT).show()
```

**执行规则**：

- 国际化支持：所有文本需定义在 `strings.xml`，严禁硬编码
- 颜色引用：必须使用 `@color/primary` 等 Design Token，严禁硬编码 `#FFFFFF`
- 尺寸引用：必须使用 `@dimen/spacing_m` 等 Design Token，严禁硬编码 `16dp`

### **3.2 工具类组织规范**

**规则**：工具类按功能域划分，归入统一位置。

**工具类型**

**归属模块**

**示例**

纯逻辑工具（无 Android 依赖）

`:core:common/util`

`DateTimeUtil`、`EncryptUtil`、`RegexUtil`

Android View 扩展

`:core:ui/extensions`

`dp2px`、`View.show()`、`View.hide()`

Android 系统相关

`:core:common/util`（需谨慎）

`AppUtil`、`DeviceUtil`

**执行规则**：

- 各 `features` 业务模块内部禁止创建 `util` 或 `helper` 包
- 发现功能重复，立即下沉（Promote）到 `core:common` 或 `core:ui`

### **3.3 日志规范**

**规则**：全项目统一使用自定义 `Logger`，禁止使用 `android.util.Log`。

kotlin

```
// ❌ 严禁使用系统 Log
import android.util.Log
Log.d("TAG", "debug message")

// ✅ 必须使用自定义 Logger
import com.yourpackage.common.logger.Logger
Logger.d(tag, "debug message")
Logger.e(tag, "error message", throwable)

// 注意：Release 包中 Logger.d() 自动不输出，仅 Logger.e() 保留
```

**执行规则**：

- 在项目根目录配置 Lint 检查，`import android.util.Log` 直接报编译错误
- `service:logger` 提供文件写入、滚动、压缩等高级功能

## **第四部分：安全与运行时铁律**

### **4.1 异常处理铁律**

**规则**：捕获异常后必须给用户明确反馈。

kotlin

```
// ❌ 严禁静默吞噬异常（Silent Failure）
try {
    apiService.login()
} catch (e: Exception) {
    Logger.e("login error", e)  // 仅打日志，用户无感知 → 绝对禁止！
}

// ✅ 必须转换为 UI 可感知的反馈
try {
    apiService.login()
} catch (e: Exception) {
    _state.value = State.Error(e.message ?: "登录失败，请重试")
    sendEvent(ShowToast("登录失败，请检查网络"))
}
```

**执行规则**：

- `BaseViewModel.execute()` 必须自动将异常转换为 `UiState.Error`
- 业务层 `catch` 块禁止 `return null` 或 `return emptyList()` 而不提示用户

### **4.2 内存与 Context 安全**

**规则**：单例中严禁持有 Activity/View 引用。

kotlin

```
// ❌ 严禁在单例中持有 Activity
object NetworkClient {
    lateinit var context: Context  // 如果传入 Activity 将导致内存泄漏
}

// ✅ 只能持有 ApplicationContext
object NetworkClient {
    lateinit var context: Application  // 通过 Application 获取
}

// ❌ 严禁在 ViewModel 中传入 Context 或 View
class LoginViewModel(context: Context) : ViewModel()  // 绝对禁止！

// ✅ ViewModel 中如需资源，传入 Application 或使用 ResourceProvider
class LoginViewModel(application: Application) : AndroidViewModel(application)
```

**执行规则**：

- `object` 单例中的 Context 必须是 `ApplicationContext`
- ViewModel 构造函数禁止传入 `Activity`、`Fragment`、`View` 或 `Context`

### **4.3 线程调度强制绑定**

**规则**：耗时操作必须显式指定线程调度器。

kotlin

```
// ❌ 严禁在主线程执行耗时操作
viewModelScope.launch {
    val data = apiService.getData()  // 未指定 Dispatcher，可能在 Main
    // 或 viewModelScope 默认是 Main，会导致 ANR
}

// ✅ 必须显式指定 Dispatcher
viewModelScope.launch {
    withContext(Dispatchers.IO) {
        val data = apiService.getData()
    }
}

// ✅ 或使用 BaseViewModel 的 execute 方法（已自动切换）
execute {
    apiService.getData()
}
```

**调度器选择规则**：

**操作类型**

**调度器**

网络请求、数据库读写、文件操作

`Dispatchers.IO`

CPU 密集型计算（JSON 解析、图片处理）

`Dispatchers.Default`

UI 更新

`Dispatchers.Main`（自动确保）

### **4.4 协程生命周期绑定**

**规则**：所有协程必须绑定到生命周期，禁止使用 `GlobalScope`。

kotlin

```
// ❌ 绝对禁止
GlobalScope.launch {
    // 页面销毁后仍在运行，浪费资源且可能崩溃
}

// ✅ 必须绑定生命周期
// ViewModel 中使用 viewModelScope
viewModelScope.launch { ... }

// Activity/Fragment 中使用 lifecycleScope
lifecycleScope.launch { ... }

// 或使用 BaseActivity 提供的 launchWhenStarted
launchWhenStarted { ... }
```

### **4.5 安全密钥隔离**

**规则**：严禁在源代码中硬编码任何密钥。

kotlin

```
// ❌ 绝对禁止！APK 可被反编译提取
private const val SECRET_KEY = "abc123"
private const val SALT = "salt_value"
private const val APP_SECRET = "your_app_secret"

// ✅ 必须通过 BuildConfig 注入（不提交到 Git）
// gradle.properties 或 local.properties 中定义
// build.gradle.kts 中配置
buildConfigField("String", "SECRET_KEY", "\"${project.properties["secretKey"]}\"")
```

**执行规则**：

- 敏感常量定义在 `local.properties`，通过 `BuildConfig` 注入
- `local.properties` 必须加入 `.gitignore`
- 服务器下发的动态 Token 必须使用 Android Keystore 加密存储

### **4.6 路由跳转健壮性**

**规则**：所有路由跳转必须经过封装，具备降级兜底逻辑。

kotlin

```
// ❌ 严禁直接调用
ARouter.getInstance().build(path).navigation()  // 路径错误时崩溃

// ✅ 必须通过 RouterHelper 封装
RouterHelper.navigate(RouterPath.HOME)  // 内部实现降级逻辑

// RouterHelper 内部实现
fun navigate(path: String) {
    val result = ARouter.getInstance().build(path).navigation()
    if (result == null) {
        // 降级处理：跳转 H5 错误页或首页
        navigate(RouterPath.ERROR_PAGE)
    }
}
```

**执行规则**：

- 所有跳转必须通过 `RouterHelper`，禁止直接使用 ARouter API
- `GlobalDegradeService` 必须处理页面未找到的场景

### **4.7 空安全规则**

**规则**：禁止使用 `!!` 强制解包（除非经过明确审查）。

kotlin

```
// ❌ 严禁使用 !!（危险操作）
val data = response.data!!  // 如果 data 为 null，直接崩溃

// ✅ 使用安全调用 + 默认值
val data = response.data ?: return
// 或
val data = response.data ?: emptyList()

// ✅ 使用 let 安全处理
response.data?.let { data ->
    // 处理非空 data
}
```

**执行规则**：

- 代码中禁止出现 `!!` 操作符，除非有 KDoc 注释明确说明原因
- 网络返回的数据必须设置默认值（空字符串、0、空列表）

## **第五部分：AI 执行协议**

### **5.1 代码生成前自检清单**

在执行任何代码生成或重构任务前，AI 必须回答以下 3 个问题：

**#**

**问题**

**检查点**

1

本次修改是否违反了层级依赖规则？

检查是否有 `core` → `features` 或 `common` → `storage` 的逆向依赖

2

本次修改是否将第三方库实现暴露给上层？

检查是否有 `MMKV`、`Retrofit` 等类出现在 `features` 的 import 中

3

本次修改是否产生了重复的轮子？

检查是否创建了与现有类功能相同的类（如新的 `PreferenceUtil`）

**如果任一答案为“是”，必须立即停止并向用户汇报替代方案。**

### **5.2 违规纠正指令**

如果 AI 生成的文件违反了上述公约，用户只需回复：

> “您违反了《Android 商业级架构 AI 开发公约》第 X 条，请立即修正。”

AI 必须：

1. 识别具体的违规条款。
2. 给出修正方案。
3. 在不破坏其他功能的前提下完成重构。

## 第七部分：编译工具链冲突预防

### 7.1 KSP 与 ViewBinding 共存规则
- KSP 处理器中**禁止**依赖 `getSymbolsWithAnnotation` 来查找标注了 `@Route` 的类，如果该类使用了 ViewBinding 作为泛型参数，该类将被静默跳过。
- **强制方案**：使用 `resolver.getAllFiles().flatMap { it.declarations }` 手动遍历所有声明，然后检查是否存在目标注解。
- 所有使用 `@Route` 的路由类，应在单元测试中验证其是否能被正确扫描到。

### 7.2 编译时序依赖声明
- 如果模块 A 的 KSP 处理器需要依赖模块 B 生成的代码，**必须**在 `build.gradle.kts` 中显式声明任务依赖关系。
- 示例：`tasks.named("kspDebugKotlin") { dependsOn(":core:ui:compileDebugKotlin") }`
